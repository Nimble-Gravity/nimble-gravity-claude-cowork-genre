Set Up & Govern

The setup sequence IT actually runs, in the order that avoids a lockout — then the one fact every regulated rollout has to address, and the architecture that compensates for it. Get this right and the rollout has air cover; get it wrong and it stalls in review.

Govern & Roll Out · ~25 min
01 — The setup sequence

Six steps, and the order matters

Cowork is an admin-enabled capability, not a free-for-all. It is off until you turn it on, so nothing is exposed before you decide to. Treat this as a checklist to confirm, not a tutorial to follow — most of it is already done at Axos, and the value here is spotting the one step that isn't.

01
Enable the capability
Turn Cowork on under Organization settings → Capabilities. Remember the ceiling rule from Lesson 2: the org toggle has to be on before a Custom role can grant Cowork to anyone.
02
Bind identity — SSO, then SCIM
SSO decides who can log in; SCIM syncs the groups your roles attach to. Verify both before enforcing RBAC. Nested IdP groups don't sync — only direct members.
03
Deploy the desktop app
Cowork needs the desktop app on macOS or Windows. Deploy through MDM — Jamf, Kandji, or Intune — or allow self-install. On Windows, use the MSIX installer, not the .exe.
04
Lock down the edges
Configure network egress allowlists, mount controls, and desktop extension allowlists in org settings, and allowlist the domains Cowork needs to reach.
05
Set the approval and folder policy
Publish the rule before people start: restricted data runs on Ask before acting, and grants go to a named project subfolder — never a drive or cloud root.
06
Set model caps — and know the spend-control limits
Model and effort caps on the Models tab are your reliable lever. Spend control is weaker than you'd expect: moving from Team to Enterprise removes several of the cost controls Team has, and there is no clean per-user spend cap. Teams needing true per-user limits are routing through an API management gateway. Plan for the cap you have, not the one you assume.
📁
Folder scope is the data control

Users select which folders Cowork may read from and write to, and finished work lands in a folder you designate. There are no granular per-file permissions — so the grant is the control. Least privilege here means picking a narrow project folder, and that single decision does more to contain risk than anything else on this page.

📡
Correction worth stating plainly: "local" does not mean the data stays put

Cowork runs on the desktop and works against local folders, and that phrasing has led people to assume the whole thing is on-machine. It isn't. The model call goes out to Anthropic's API. Whatever Cowork reads from a granted folder to answer a request leaves the machine to be processed — the files are local, the reasoning over them is not. That is precisely why the folder grant is an access-control decision and not just a convenience setting: scope it as though everything inside it is being sent out, because functionally it can be.

02 — The audit-coverage gap

The one fact every regulated rollout must address

Be precise about this — it is the first thing an examiner will ask and the fastest way to lose credibility if you soften it. Verified against Anthropic's documentation on 21 July 2026.

The gap, in Anthropic's own words

Cowork isn't in the audit surfaces

Anthropic's Cowork admin guide states plainly: "the Compliance API and Audit Logs do not cover Claude Cowork yet." Cowork activity is excluded from Audit Logs, the Compliance API, and Data Exports — on every tier, Enterprise included — and conversation history lives locally on the user's machine.

Anthropic shipped the Compliance API on 21 May 2026. Cowork is simply not in its scope yet.
Why you manage it rather than wait

A known limitation, not a blocker

Anthropic is actively closing this. The defensible position is not "we waited" — it is "we knew, we compensated, we wrote it down, and we re-check it every cycle." That sentence is what the capstone one-pager has to earn.

Name the gap yourself, before the examiner does.
03 — What compensates for it

Three planes of monitoring

No single surface covers a Cowork deployment. A complete audit story layers three, and a shared user account identifier lets you correlate across them.

01
Control plane — Compliance API
Org-level administrative activity, with 28 security and compliance integrations including Microsoft Purview, Okta, CrowdStrike, Wiz, and Zscaler. Covers the tenant — not Cowork sessions.
02
Agent plane — OpenTelemetry
The closest thing to session-level visibility you have today: tool and connector calls, files read and modified, skills used, and whether each action was approved manually or auto-initiated. Covered in Lesson 4.
03
Network plane — proxy or gateway
An on-device proxy or LLM gateway captures what the other two miss at the network and tool boundary. This is your DLP hook, since native DLP alerting on Cowork conversations isn't available.
🛡️
Route the work that can't wait for the gap to close

Some workloads need zero data retention or centralized audit today. Those don't belong in the Cowork interface — route them to Anthropic's audited surfaces, the API or Claude Code Enterprise. Deciding this per workload, in writing, with a named example, is what turns "we're aware of the gap" into a control.

For a Microsoft-stack bank there is a stronger option worth naming: Claude on Azure AI Foundry. It is a materially better posture for high-assurance work because the infrastructure is genuinely yours — real components you install and configure inside your own boundary — rather than a setting you toggle. The trade is effort: it is a deployment project, not a checkbox. That same gateway layer is also where per-user spend control becomes possible.

04 — Data and retention

What's kept, for how long, and what isn't

The questions a CISO asks first. The commercial answer is reassuring, with one boundary worth stating plainly so it doesn't surface later as a surprise.

No training, 30-day retention

On commercial plans — Team, Enterprise, and API — Anthropic doesn't train on your data by default, and retention is 30 days.

ZDR has a boundary

Zero Data Retention is available for the API and Claude Code Enterprise, but not for the Team or Enterprise Cowork interface. State that limit up front for regulated workloads.

📄
What to put in front of your CISO

One page: Cowork is admin-enabled, identity-bound through SSO and SCIM, and MDM-managed; administration is delegated through Custom roles rather than blanket Owner; access is least-privilege by folder with approvals on for restricted data; commercial data isn't trained on and is kept 30 days; the audit-coverage gap is named, compensated by the three monitoring planes, and re-verified each cycle; and high-assurance workloads are routed to audited surfaces. That is the governance case the Control Room asks you to make — and the acceptance checklist scores you against exactly these points.
