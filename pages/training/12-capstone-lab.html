Status Badges — Icon + Label, Never Color Alone

Feature 2
Status Badges
~20 min

Replace the plain status text in the table with styled badge elements. Each badge shows an icon and a label — never color alone. This is a WCAG 2.1 AA requirement from your CLAUDE.md: colorblind users must be able to read status without relying on color.

Ahead: ↑ green badge. On Track: → teal badge. At Risk: ↓ amber/orange badge.

Feature 3 of 3

Add Row Modal — Form with Validation

Feature 3
Add Row Modal + Form Validation
~25 min

Add an "Add Branch" button that opens a modal dialog with a form. The form collects Branch ID, Branch Name, Target Approvals, Actual Approvals, and Last Updated. Status is computed — not entered. Validation ensures no field is empty and that approvals are positive numbers.

This is the most complex feature. Use Plan mode. Read the plan carefully. Expect to iterate once on the validation logic.

Workshop Complete

You built a real product with AI as your pair programmer.

In six hours, you went from zero to a functioning interactive dashboard — using Claude.ai for design thinking and Claude Code for implementation. More importantly, you have a set of durable habits:

Prompt structure
Role → Task → Context → Format. Every high-stakes prompt. Every time. The habit that produces the biggest quality jump per unit of effort.
CLAUDE.md as the contract
Your standing instructions, maintained over time. The delta between a team with a maintained CLAUDE.md and one without is visible in every commit.
Plan mode before execution
Read the plan. Check files, scope, approach. Correct before the edit, not after. Ten seconds now saves thirty minutes later.
/clear between tasks
Context hygiene is not optional. Long sessions with accumulated context produce drift. Clear the slate and start each new feature clean.
Read the diff before committing
The diff is the review. Audit it like a PR from a junior developer. You own the git history — not Claude.
Feature-by-feature strategy
One coherent task. One session. One clean commit. This is what separates teams that scale AI usage from teams that create maintenance debt.
