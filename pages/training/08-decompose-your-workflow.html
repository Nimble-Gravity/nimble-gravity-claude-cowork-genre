Decompose Your Workflow

A skill is a reusable instruction set that fires automatically when the situation matches — not a prompt you type each time. This lesson starts by defining what a skill actually is, then shows you how to map the workflow you will turn into one: steps, data sources, and a clear picture of good versus bad output.

Build a Skill · ~20 min
00 — What is a skill

A reusable instruction set that fires automatically

In Workshops 1 and 2 you typed a prompt to describe each task. A skill changes that: it holds the task description, and Cowork reads it at startup. When the situation matches the skill's description, Cowork routes to it — you do not have to describe the task every time.

Prompt vs skill

A prompt you type each time is a one-off delegation — you describe the task, Cowork runs it, done. A skill is that same instruction set installed once. Cowork reads the skill's description at startup and selects it automatically when you ask for a matching task. You just work; Cowork routes to the right tool.

The file

A skill is a folder containing one file: SKILL.md. YAML frontmatter at the top gives it a name and a description. A Markdown body below holds the instructions. The same file runs in Claude Cowork and Claude Code on the open Agent Skills standard — one skill, both surfaces.

↻
The Workshop 2 through-line

In Workshop 2 you refined a prompt over a few sessions until it reliably worked. A skill is that prompt packaged — so anyone on your team gets the same quality result without having to rediscover the right words. That is what this workshop builds.

01 — Map the workflow

Get the job out of your head and onto a page

Use a simple canvas to write the workflow down. This is the structure Nimble Gravity walks clients through, and it lines up with how Anthropic recommends approaching agentic work.

Use case and challenges

What are you trying to produce, and what makes it slow or error-prone today? Name the deliverable and the pain in a line or two each.

Process, outcomes, data

List the steps you take, the result you're after, and the data source each step needs. That gives you a plan to build the skill from.

02 — Work in slices

Build one slice at a time

Anthropic's guidance on long-running agents warns against trying to do a big task in one shot. You run out of context, or a later step decides the job is done before it is. Splitting the work into small slices, each leaving a clear artifact, avoids both.

Pick the first slice

The smallest useful piece

Like building a proposal one slide at a time, start with the first slice that delivers something real. Get it working, then move to the next instead of trying to cover everything at once.

Each slice should leave behind an artifact you can check.
Name the data per slice

Where does the input come from?

For each slice, write down the source: a folder, a connector, a system. A skill that knows where its inputs live is a skill that works the first time.

Vague inputs are the most common reason a skill underperforms.
03 — Define good output first

Decide what good looks like before you build

This is Anthropic's evaluation-driven method in plain terms. Write down what a good result and a poor one look like before you write the skill, so you can tell whether it's working.

01
Show good and bad output
Write a short example of a strong result and a weak one. These become the examples your skill follows and the cases you test it against.
02
Run it without a skill first
Try the task in plain Cowork to see where it falls short. The gaps tell you what the skill needs to add, and nothing more.
03
Three scenarios, one baseline
Anthropic recommends at least three realistic test scenarios and a baseline before you write instructions. That's the rubric for the next lesson.

What you should have by the end

A one-page canvas: the use case, the challenges, the steps, the outcomes, the data sources, and an example of good versus bad output. Lesson 2 turns that into a working SKILL.md.
