Anatomy of a Skill

A skill is a folder with a SKILL.md file: YAML frontmatter and a Markdown body, on the open Agent Skills standard. Anthropic has clear opinions about how to write one. Follow them and your skills fire when they should and stay easy to maintain.

Build a Skill · ~20 min
01 — Frontmatter

The description does most of the work

At startup, Claude only loads each skill's name and description. That's what it reads to choose the right skill out of dozens, so a clear description is what gets the skill to fire when it should.

name

Up to 64 characters, lowercase letters, numbers, and hyphens, and no reserved words like "anthropic" or "claude." Anthropic suggests the gerund form, such as processing-pdfs or analyzing-spreadsheets.

description

Third person, up to 1024 characters. Say what the skill does and when to use it, with the words that should trigger it: "...Use when working with PDF files or when the user mentions PDFs, forms, or document extraction."

02 — The body

"Concise is key. The context window is a public good."

That's Anthropic, word for word. Add only the context Claude doesn't already have, and keep the file short so it stays fast to read and simple to maintain.

Keep it under 500 lines

Use progressive disclosure

Treat SKILL.md as a table of contents that points to detail files Claude opens only when it needs them. Keep references one level deep, since Claude may only partly read nested ones, and add a table of contents to any reference file over 100 lines.

Split the body into separate files as it nears 500 lines.
Explain the why

Don't fall back on ALL-CAPS

Instead of MUST, ALWAYS, and NEVER, state the rule and then say why, so Claude can generalize from it. Give a few real input and output examples, offer a sensible default with a way out, and use forward-slash paths.

Make instructions more specific as the task gets more fragile.
03 — Evaluation-driven development

"Create evaluations before writing extensive documentation."

Anthropic's words again. You build a skill by testing it, not by writing prose and hoping. This is the loop your lab runs next.

01
Evals first, then minimal docs
Find the gaps by running without the skill, write at least three test scenarios, set a baseline, then add only the instructions needed to pass.
02
Test across models
Run your tests on Haiku, Sonnet, and Opus. A skill that only works on the largest model isn't finished.
03
Write with one Claude, test with another
One Claude helps you draft the skill; a fresh Claude uses it cold. There's no built-in eval runner, so you put together a simple one.
📐
Use the doc as your rubric

These standards come from Anthropic's skill-authoring best-practices doc on platform.claude.com, current as of June 2026. Use it as the grading rubric for every skill you ship, and check it again before each cohort, since it changes.
