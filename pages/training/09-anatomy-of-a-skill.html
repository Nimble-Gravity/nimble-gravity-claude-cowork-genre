Anatomy of a Skill

A skill is a folder with a SKILL.md file: YAML frontmatter and a Markdown body, on the open Agent Skills standard. Anthropic has clear opinions about how to write one. Follow them and your skills fire when they should and stay easy to maintain.

Build a Skill · ~20 min
01 — Frontmatter

The description does most of the work

At startup, Claude only loads each skill's name and description. That's what it reads to choose the right skill out of dozens, so a clear description is what gets the skill to fire when it should.

name

Up to 64 characters, lowercase letters, numbers, and hyphens, and no reserved words like "anthropic" or "claude." Anthropic suggests the gerund form, such as processing-pdfs or analyzing-spreadsheets.

description

Third person, up to 1024 characters. Say what the skill does and when to use it, with the words that should trigger it: "...Use when working with PDF files or when the user mentions PDFs, forms, or document extraction."

02 — The body

"Concise is key. The context window is a public good."

That's Anthropic, word for word. Add only the context Claude doesn't already have, and keep the file short so it stays fast to read and simple to maintain.

Keep it under 500 lines

Use progressive disclosure

Treat SKILL.md as a table of contents that points to detail files Claude opens only when it needs them. Keep references one level deep, since Claude may only partly read nested ones, and add a table of contents to any reference file over 100 lines.

Split the body into separate files as it nears 500 lines.
Explain the why

Don't fall back on ALL-CAPS

Instead of MUST, ALWAYS, and NEVER, state the rule and then say why, so Claude can generalize from it. Give a few real input and output examples, offer a sensible default with a way out, and use forward-slash paths.

Make instructions more specific as the task gets more fragile.
03 — Evaluation-driven development

"Create evaluations before writing extensive documentation."

Anthropic's words again. You build a skill by testing it, not by writing prose and hoping. This is the loop your lab runs next.

01
Evals first, then minimal docs
Find the gaps by running without the skill, write at least three test scenarios, set a baseline, then add only the instructions needed to pass.
02
Test across models
Run your tests on Haiku, Sonnet, and Opus. A skill that only works on the largest model isn't finished.
03
Write with one Claude, test with another
One Claude helps you draft the skill; a fresh Claude uses it cold. There's no built-in eval runner, so you put together a simple one.
📐
Use the doc as your rubric

These standards come from Anthropic's skill-authoring best-practices doc on platform.claude.com, current as of June 2026. Use it as the grading rubric for every skill you ship, and check it again before each cohort, since it changes.

04 — A worked example

A small, finished skill: memo generation

Read a real one against the rules above. The description is keyword-rich and third person, the body is well under 500 lines, it carries a good-versus-bad example and a guardrails block, and it points one level deep at a format file. Analysts lose hours to memos, so the payoff is immediate.

memo-generation / SKILL.md 
---
name: memo-generation
description: Drafts a structured business memo from rough source material — meeting notes, an email thread, or a few bullet points — in the team's standard format and voice, leaving a clear draft for a person to review and sign off. Use when someone asks to "write a memo", "turn these notes into a memo", "draft a summary memo", or "put this in memo form", especially for recurring internal or client-facing memos in regulated, document-heavy work.
---

# Memo Generation

Turn rough input into a clean, well-structured memo draft. Analysts and associates lose
hours to this; the structure rarely changes, so it is an ideal first skill. The output is
always a draft for a human to review — never a final, never sent automatically.

## How to draft

1. Read everything first. Understand the whole input before you summarize.
2. Lead with the bottom line. The first sentence states the conclusion or the ask.
3. Use the standard sections: Purpose, Background, Key Points, Recommendation / Next Steps.
   If no format is given, use reference/memo-format.md and say you used the default.
4. Cite the source for any figure or claim — name the file, email, or note.
5. Flag gaps, don't fill them. Unsupported section -> [NEEDS INPUT: ...].
6. Match the house voice if a voice-profile exists; otherwise keep it plain and tight.

## Good vs. bad output

Good — bottom line first; every claim traceable to a source; gaps flagged; one page;
ready for a reviewer to approve with light edits.

Bad — buries the ask in paragraph three; invents a number that isn't in the source;
fills an empty section with filler; reads like a chat transcript; needs a full rewrite.

## Guardrails

- Never present the memo as final or send it anywhere. It is a draft for review.
- Never fabricate figures, names, or quotes. Unknown -> [NEEDS INPUT: ...].
- Keep it to one page unless the user asks for more.

05 — Good skill vs. bad skill

The same rules, side by side

Held against Anthropic's standards, the difference between a skill that fires reliably and one that doesn't is usually visible in the first ten lines. The memo skill above is the good column.

Good

Triggers, and stays out of the way

Third-person description packed with trigger words; a body under 500 lines; one clear good-and-bad example; a default with an escape hatch; detail pushed one level deep into a reference file.

A fresh Claude picks it up and uses it correctly, cold.
Bad

Under-triggers, over-explains

First-person or vague description that never fires; a 900-line body of ALL-CAPS rules; no examples; nested references three deep; no evals, so nobody knows whether it works.

It reads like documentation nobody tested.
