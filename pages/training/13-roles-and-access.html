Roles & Access

Who can turn Cowork on, who can see the analytics, who can approve a connector, and who can remove the plugin Legal made mandatory. RBAC is the control that makes a phased rollout possible — enable the pilot group, not the org. This is the lesson your IT partner came for.

Govern & Roll Out · ~20 min
01 — The roles you start with

Owner is not an admin convenience

Claude Enterprise gives you three static roles plus a single Primary Owner. The trap is using Owner as a shortcut for "needs to administer something" — because an Owner cannot be narrowed.

User · Admin · Owner

Three static roles, plus one Primary Owner per organization. Billing and Developer are also assignable. Anthropic's wording is absolute: "Owners and Primary Owners always have full access to all features."

Custom roles are the real tool

On Enterprise, build Custom roles at Organization settings → Roles. A Custom role grants a named set of capabilities and admin areas to a group — which is how you delegate one job without handing over everything.

🔑
The test to apply

For every person holding Owner, ask: do they administer the whole organization, or one area of it? If the honest answer names a single area — billing, or usage reporting, or adding and removing members — they belong on a Custom role instead. In the Control Room lab you will find a role matrix where four of six Owners fail exactly this test.

02 — Delegating administration

Six admin areas, three levels each

Custom roles carve up administration into six areas. Each one is set to No access, Can view, or Can manage — so "she pulls the usage numbers for the steering committee" is a role, not an Owner seat.

The six areas

Identity, Billing, Analytics, Privacy, Users, Libraries

Identity & Access covers editing roles and groups. Billing covers payment and plan. Analytics covers usage reporting. Privacy covers privacy settings. User Management covers inviting and removing members. Libraries covers shared library content.

Each area: No access · Can view · Can manage.
The misconfiguration everyone makes

Custom roles inherit nothing

Anthropic's exact words: "Members assigned to custom roles don't automatically inherit organization-enabled capabilities. Every capability a 'Custom' role member needs must be explicitly granted." A Custom role with no capabilities listed grants none — even with Cowork enabled org-wide.

The org toggle is the ceiling. The role is the grant.
03 — What a role actually controls

Capabilities, connectors, and models

Three tabs decide what a group can reach. The Models tab is the one people forget — and it is the structural answer to a runaway model bill.

01
Capabilities
Per-feature toggles: Cowork, Claude Code, Web Search, Memory, Projects, Code Execution. This is what makes a phased rollout possible — enable Cowork for the pilot group and nobody else.
02
Connectors
Each connector is Always allow, Needs approval, or Blocked, with per-tool granularity via Custom. Anything reaching cedent-confidential or claims data belongs on Needs approval. Note the limit: connector settings are organization-wide — there is no per-group connector control, so the strictest requirement governs everyone.
03
Models
Enable or disable specific models, cap effort levels, and set the default model for new conversations. When the analytics show one desk running everything on Opus, this tab is the fix — a cap, not a memo.
04 — Groups, sync, and permission math

Two rules that surprise everyone

Roles attach to groups, and groups come from your identity provider. Both of the rules below have cost real pilots a morning.

Nested IdP groups don't sync

Groups are created manually or synced via SCIM, and are managed at the parent organization. But only direct members synchronize — a group that inherits its membership from a parent group in your IdP will arrive empty, and nobody gets the capability.

Capabilities add, spend limits don't

Capabilities are additive — a user in several groups gets the union of their permissions. Where group spend limits exist they work the opposite way: the most restrictive limit wins. A second group never loosens a cap. Caveat from the field: Enterprise removes several cost controls that Team has, and there is no clean per-user spend cap — teams that need one are routing through an API management gateway.

🧭
Sequence it: SSO first, then RBAC

SSO controls who can log in; RBAC controls which product surfaces they reach. Configure and verify SSO and role assignment before you enforce RBAC — enforcing roles against an identity source that hasn't settled is how administrators lock themselves out of their own tenant.

05 — Distributing plugins by group

"Available" is not a control

Private-marketplace plugins carry an install preference, and Enterprise supports group-level overrides tied to your SCIM groups. If a plugin carries something Legal requires, the preference is the enforcement.

Four install preferences

From Installed by default to Not available

Required — automatically installed for all members, without the option to remove it. Installed by default — automatically installed, but members can uninstall. Available for install — listed in the catalog, members self-install. Not available — hidden from the catalog entirely. A mandatory disclaimer or review gate has to be Required; anything else is a suggestion.

Set per-group via the Custom access column → Add groups.
The edit model

Owner pushes, members copy

Members cannot edit an organization-managed plugin; if they edit files locally, Cowork warns them before an update overwrites the change. To customize, a member copies a template and tweaks their own version — and good changes get folded back into the managed one.

One source of truth; everyone else forks from it.
⚠️
"Not available" is not a block — the most permissive group wins

When a member belongs to several groups with different install preferences, the most permissive setting applies, resolved in this order: Required → Installed by default → Available for install → Not available. So hiding a plugin from one group achieves nothing if that person is also in a group where it's Required or Available. If a plugin genuinely must not reach someone, the control is their group membership, not the preference. Changes also land on each member's next session or plugin refresh — not instantly.
