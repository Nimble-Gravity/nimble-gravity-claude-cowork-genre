# Workflow Decomposition

Help the user turn a fuzzy, in-their-head workflow into a clear blueprint, then offer
to draft it as a skill. The interview does the teaching: by answering, the user learns
what a skill needs before they ever write one.

## The interview

Work one slice at a time. Ask, reflect, then go deeper.

1. The job. What is this workflow, in one sentence? When do you reach for it?
2. The trigger. What words or situation should make Cowork pick this up later?
   (This becomes the skill's description — capture it precisely.)
3. The steps. "What do you do first? Then what? Then what?" Walk the whole sequence.
4. The data. For each step, what does it read or need — which folder, file, or system?
5. Good output. Describe one example of a great result. What makes it great?
6. Bad output. What does a wrong or weak result look like? What must never happen?

## Summarize the blueprint

Play the workflow back: job and trigger (one line each), numbered steps with the data
source on each, the definition of done, and the guardrails. Ask the user to correct it.

## Offer to draft the skill

Once the blueprint is confirmed, ask: "Would you like me to draft this as a skill?"
If yes, write a tight third-person description from the job and trigger, turn the steps
into the body (under 500 lines, references one level deep), fold in the good and bad
examples, and hand off to the skill-creator loop for evals — evals before docs.

Don't over-build. If a step is genuinely one instruction, leave it as one line.

02 — Capture intent and draft

Begin with the trigger

The skill-creator opens by capturing intent, and the question that matters most is when the skill should fire. Your answer to that drives the description.

Answer the intent questions

What does it do, when should it fire, what inputs does it need, and what does good output look like? Pull these straight from your Lesson 1 canvas.

Draft the SKILL.md

Write a gerund name, a third-person description full of trigger words, and a body under 500 lines with one example of good output. Keep it minimal and let the evals tell you what to add.

03 — Test and improve

Run the skill against a baseline

Write two or three realistic test prompts, run them with the skill and without it, and grade the difference. Then tune the description so the skill actually triggers.

Run the loop

Draft, test, improve, repeat

The skill-creator runs your tests in parallel, one with the skill and one baseline without, drafts objective checks, grades them, and shows the results in a review viewer. Keep going until it passes.

Write a pushy description; Claude tends to under-trigger skills.
Tune the description

Choose by held-out test score

The run_loop optimizer splits your evals 60/40, runs each query a few times for a steady trigger rate, and picks the best description by the held-out test score so it doesn't overfit.

Heads up: this optimizer needs a raw ANTHROPIC_API_KEY; SSO-only users can't run it.
04 — Package and share

Produce a .skill, then take it home

Package the skill, understand how sharing works, and leave with homework: build the skill for a real workflow of your own.

01
Package to a .skill
The skill-creator wraps your folder into a single .skill file. It's portable and ready to share or reuse across your team.
02
Know your sharing scope
On Team and Enterprise, skills sit under Customize, then Skills, as Personal, Shared, or Organization. Shared skills are read-only and update on their own. Sharing stays off until an owner turns it on.
03
Homework
Build one skill for a recurring task from your own job, using the template you just shipped. Bring it to office hours.
🎒
Keep your skills tool-agnostic

Build skills in the Claude Code plugin structure, since it's the superset, and keep the SKILL.md free of tool-specific assumptions. The same file runs in Claude Cowork and in Claude Code, so one skill serves both.
