Artifacts & Output Types

Claude can generate more than text. Artifacts are live, interactive outputs — code you can run, documents you can edit, diagrams you can view — all inside the conversation. Knowing which output type to request determines how useful the result is.

Claude.ai · ~20 min
01 — Artifacts explained

An Artifact is an output Claude separates from the chat.

When you ask Claude to write code or create a document, it can render the result in a dedicated panel next to the chat — an Artifact. You can preview it live, copy it, or ask Claude to modify it without re-generating the whole response. Artifacts are the iteration surface.

💻
Code
Runnable
HTML/CSS/JS renders live in the preview panel. Other languages (Python, TypeScript, etc.) display with syntax highlighting and copy-to-clipboard. The preview panel updates immediately when you ask Claude to make changes.
Use for: UI prototypes, interactive components, data utilities, scripts.
📝
Document
Editable
Markdown documents render with formatting in the preview panel. Claude can append to, rewrite sections of, or restructure the document as separate operations without regenerating the whole thing.
Use for: Requirements docs, changelogs, meeting notes, Jira descriptions, README files.
🔀
SVG / Diagram
Visual
SVG renders directly. Mermaid diagrams (flowcharts, sequence diagrams, ER diagrams) are supported via code artifacts. Useful for architecture diagrams, user flow maps, and data model visualization.
Use for: System diagrams, user flows, data models, quick wireframe sketches.
💡
How to request an Artifact

Claude creates Artifacts automatically when the output is clearly a standalone artifact (code, a document, SVG). You can also ask explicitly: "Create a code artifact" or "Put this in an Artifact so I can iterate on it." If Claude inline-pastes code instead of creating an artifact, try: "Rewrite that as an Artifact."

02 — Iteration pattern

One prompt to create. Many messages to refine.

The power of Artifacts isn't the first output — it's the iteration loop. Because the Artifact is in the same context as the chat, you can refine it with natural language follow-ups. Each message updates the Artifact in place.

1
Create the Artifact
"Create an HTML/CSS/JS Artifact for a Loan Application status table. Use this data: Branch Springfield, 90 target approvals, 87 actual, On Track. Branch Riverside, 75 target, 81 actual, Ahead. Use a clean table layout with a header row."
2
Refine the output
"Update the Artifact: add a Status column. Each status badge should use an icon + text label. Ahead = ✓, On Track = →, At Risk = ⚠. Use green/blue/amber colors but the icon is mandatory — do not rely on color alone."
3
Add interactivity
"Add filter buttons above the table for each status value. Clicking a button filters the visible rows. Include an All button that resets the filter. Keep the CSS inline, no external libraries."
4
Export and use
Copy the Artifact source or open it in a new tab to share with stakeholders. At this point you have a working interactive prototype — built in 3 prompts, zero code written by hand.
03 — When to use Artifacts

Artifacts for standalone outputs. Chat for decisions.

Not everything should be an Artifact. Chat is the right surface for reasoning, analysis, Q&A, and planning conversations. Artifacts are the right surface for things you'll iterate on or hand off.

Use an Artifact

Standalone outputs you'll iterate or share

  • Prototypes and interactive components
  • Documents you'll export or paste into Confluence
  • Requirement lists, acceptance criteria, checklists
  • Architecture diagrams and user flows
  • Scripts and utilities you'll run externally
Keep in chat

Reasoning and analysis

  • Explaining a concept or decision
  • Answering a question about the codebase
  • Reviewing something for issues or gaps
  • Planning a sequence of steps before you start
  • Comparing options or trade-offs
04 — Sharing & Collaboration

Artifacts are the fastest stakeholder prototype in your toolkit.

An HTML Artifact can be opened in a browser tab and shared as a URL. For teams doing stakeholder demos before components exist in the real UI, this is genuinely useful — a working, interactive prototype you can show the same day you write the requirement.

What makes a good stakeholder Artifact

Realistic data (use actual LOAN-2024-Q3 branch names, not "Branch A / Branch B"). Correct status values. Functioning interactions (filter, sort, form submission). Close enough to the real component library's visual style that the stakeholder is reviewing the right thing.

What it is not

An Artifact is not production code. It's not an Angular component. It's a communication tool — fast, disposable, purpose-built for alignment. Don't mistake it for implementation. Use it to align on what you're building, then build the real thing in Module 3 and 4.
