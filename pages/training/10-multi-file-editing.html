Multi-File Editing & Iteration

Real features touch multiple files. This lesson covers how to coordinate Claude Code across files, how to break tasks down when a request is too large, how to read errors and iterate, and the "too big to fail" trap that kills agentic sessions.

Claude Code · ~20 min
01 — The Multi-File Reality

Real features don't live in one file.

A filter button row in your loan dashboard means changes in at least three places: HTML structure for the buttons, CSS for the active state and layout, and JavaScript for the filter logic. Claude Code handles all three — but you have to understand the coordination pattern to use it well.

Under-scoped (misses files)
Incomplete scope 
Add the filter bar to @index.html

Claude adds the HTML but doesn't know to update the CSS or JS. The buttons render but don't work and have no visual state. You ask again, and again. Three prompts for one feature.

Correctly scoped (whole feature)
Complete scope 
Add the filter bar as a complete feature across all three files:
@index.html — add the filter-bar div with three buttons
@styles.css — add .filter-bar layout and .filter-btn active state
@dashboard.js — add filterByStatus() and wire up the buttons

When a filter is active, show only rows matching that status.
"All" shows every row.

One prompt. Claude touches all three files. Buttons work and have visual state. Feature is complete in one pass.

🧩
The complete-feature prompt pattern

When building a feature, list each file it needs to change and what that change is. This prevents Claude from under-scoping — which leads to multiple correction prompts that add up to more tokens than one well-scoped prompt would have used.

02 — Task Sizing

Not too big. Not too small. One coherent unit.

The size of a task matters. Too small and you're doing the coordination work yourself. Too large and Claude loses coherence mid-execution — it forgets constraints from CLAUDE.md, makes inconsistent choices across files, or produces implementations that are technically correct but architecturally incoherent.

Fix a typo
One prompt
Add filter buttons
One prompt
Add modal form
Use Plan mode
Refactor entire JS file
Break into parts
Rebuild entire app
Don't do this
⚠️ Too Big to Fail Trap
One prompt for everything

"Build me the complete Loan Dashboard with filters, badges, a modal form, validation, animations, and export to CSV."

Claude starts strong, loses coherence around file 3, makes assumptions that contradict CLAUDE.md, produces code that technically runs but is hard to maintain. You spend more time fixing than it would have taken to write smaller prompts.

✅ Feature-by-Feature Strategy
One complete feature per session

Build the table. Test it. Commit. /clear. Build the filters. Test them. Commit. /clear. Build the modal. Each feature starts from a clean context with only the current task.

Slower in theory. Faster in practice. Each feature is focused, coherent, and reviewable as a unit. The git history is clean.

03 — Reading Errors & Iterating

Claude Code fixes its own errors — if you give it the error.

When something breaks, don't describe the error in prose. Paste the actual error message. Claude Code is trained on thousands of error formats — browser console output, TypeScript errors, test failures — and it can usually fix them in one turn when given the raw text.

Error iteration example
Add the filter functionality per our last plan

Done. I've updated all three files.

Uncaught ReferenceError: filterByStatus is not defined
at HTMLButtonElement.onclick (index.html:34:32)

Getting this error: [paste above]

The function is in dashboard.js but the script tag loads after the inline
onclick in the HTML. Moving the onclick to a DOMContentLoaded listener in
dashboard.js so the function is available when the buttons are clicked.

✓ Updated dashboard.js — moved event binding to DOMContentLoaded
📋
Paste the full error, not a summary
"It's not working" forces Claude to guess. "Uncaught TypeError: Cannot read properties of null" gives Claude the exact failure mode. Always paste the raw error output.
🔢
Include line numbers when they appear
Browser error messages include file name and line number. Include them. Claude Code can use them to go directly to the problem location without searching.
🔄
If the fix creates a new error, paste that too
Error iteration is a normal part of the workflow. Each cycle reduces the problem. Two or three cycles to a working feature is standard — it's not a sign something is wrong.
🛑
If errors persist after 3 cycles, /clear and restate
Three error cycles on the same issue usually means the accumulated context is confusing Claude. Clear, restate the feature and the current broken state fresh, and try again.
