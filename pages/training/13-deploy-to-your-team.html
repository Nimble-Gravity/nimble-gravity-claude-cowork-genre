Deploy to Your Team

Go from a plugin you built to a plugin your team runs. Team and Enterprise owners set up a private marketplace, choose install policies, and use a clean edit model: the owner pushes updates, and members copy and tweak their own versions.

Plugins & Rollout · ~20 min
01 — Stand up a private marketplace

Your plugins, in one place for the org

Team and Enterprise owners can publish a private marketplace two ways. Choose based on how often you iterate.

ZIP upload

Quick to iterate: upload a packaged plugin directly. Good while you're still refining it.

GitHub sync

Version-controlled from a private repo, with syncs that can take up to half an hour. Two limits to keep in mind: 100 plugins, and 50 MB per file.

02 — Set the install policy

Decide what's required and what's optional

Per-plugin preferences control how a plugin reaches people, and Enterprise can override them per group.

Four install modes

Required through Not available

Installed by default lets members uninstall, Required auto-installs and can't be removed, Available is self-service, and Not available hides it. Match the mode to how essential the plugin is for that group.

Enterprise can override these preferences per group.
Owner pushes, members copy

How edit rights work

Members can't edit an organization-managed plugin. If someone edits the files locally, Cowork warns them before an update would overwrite the changes. To customize, a member copies a template, through the "Plugin Create" plugin, and tweaks their own copy.

One source of truth; everyone else forks from it.
03 — Run it over time

A deployed plugin needs an owner and a rhythm

Team deployment isn't a one-time push. It's a managed asset, and a few habits keep it healthy.

01
Name an owner
The owner's updates roll out automatically to everyone on the managed plugin. Without an owner, it drifts out of date.
02
Schedule team work
Pair deployed plugins with scheduled tasks for recurring team jobs, and remember Claude Cowork's awake-and-open limit when you do.
03
Collect what members improve
When a member copies and improves a plugin, fold the change back into the managed version. That's how the library gets better over time.
🚀
This is where it scales

Personal skills prove the value; managed plugins spread it. Once a team is forking and improving a shared plugin, you've moved from one person's productivity to a capability the organization owns, which is what the governance and adoption work in the capstone is there to protect.
