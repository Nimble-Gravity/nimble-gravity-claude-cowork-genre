Explore, Plan & Context Hygiene

The tools that make Claude Code precise: @ references to point Claude at specific files, Plan mode to see the plan before the edit, and /clear to prevent context drift between unrelated tasks.

Claude Code · ~20 min
01 — @ File References

Pointing Claude at the right files is the biggest quality lever.

Claude Code has access to your entire codebase, but that doesn't mean it reads everything on every prompt. Without guidance, it guesses which files are relevant. With @ references, you tell it exactly what to look at — and the output is dramatically more accurate.

Without @ references
Vague task 
Add sort behavior to the loan table.

Claude searches the codebase, might find the wrong table component, guesses the sort pattern, and writes code that doesn't match your existing service layer. You fix the mismatches.

With @ references
Targeted task 
Add column sort to @loan-table.component.html.
The sort state goes in @loan-table.component.ts.
Match the pattern in @branch-filter.component.ts for how we handle state changes.

Claude reads exactly the files you specified. It understands the existing pattern, writes code that matches your naming conventions, and puts state management in the right place.

📂
@ works for directories too

@src/components/ reads the full contents of the components directory. Use this when you want Claude to understand an entire subsystem before making changes. Be aware: large directories add significant tokens to the context.

02 — Plan Mode

See the plan before the edit. Always.

Plan mode (Shift+Tab to toggle) puts Claude Code in a "think before act" state. It reads files, forms a plan, and shows you exactly what it intends to do — then waits for your approval. This is the single most important Claude Code habit to build.

Claude Code — Plan Mode enabled
Add status filter buttons above the loan table. Ahead / On Track / At Risk. Use @loan-dashboard.component.html and @loan-dashboard.component.ts

Reading loan-dashboard.component.html... ✓
Reading loan-dashboard.component.ts... ✓

📋 Plan (Plan mode — no changes will be made until approved):

1. Add a <div class="filter-bar"> above the table in the HTML template
2. Create three <button> elements inside it, using mat-button directive
3. Add an activeFilter string property to the component class
4. Add setFilter(value: string) method that updates activeFilter
5. Add filteredLoans getter that filters loanData based on activeFilter
6. Update template to use filteredLoans | async in the *ngFor

Files to be modified: loan-dashboard.component.html, loan-dashboard.component.ts
No new files will be created.

Approve this plan? (y/n/edit) _
When to use Plan mode

Any task touching more than 1–2 files

New features, refactors, anything where the scope might be larger than it looks. The 10 seconds it takes to read the plan is always cheaper than unwinding a bad implementation.

Toggle: Shift+Tab to enable Plan mode. Shift+Tab again to disable.
Reading the plan

Check: files, scope, approach

Is Claude touching the right files? Is the scope right (not too many files, not too few)? Does the approach match your patterns? If anything looks wrong — edit the task before approving.

Type 'edit' to modify the plan before execution. Never just approve blindly.
03 — Context Hygiene

Long sessions drift. /clear is your reset.

Every message in a Claude Code session adds to the context window. After 20–30 exchanges, especially across different tasks, the accumulated context starts to interfere with new requests. Claude will reference earlier decisions, earlier file states, and earlier errors that no longer apply. /clear resets to a clean slate.

Signal to /clear
Starting a different task
You finished adding the filter. Now you want to add the sort. New task, new files, new context. The filter conversation is noise now.
Best practice: /clear between unrelated features, even in the same session.
Signal to /clear
Claude is "stuck"
Claude keeps referencing an error from 15 messages ago that you already fixed. Or it keeps suggesting the same wrong approach. Context drift.
/clear + restate the task cleanly. You'll often get a better response immediately.
Signal to /clear
Session over 30 exchanges
Long sessions accumulate a lot of file contents, intermediate states, and decisions that are no longer relevant. Quality tends to degrade past ~30 turns.
Rule of thumb: /clear every 20–30 exchanges on long work sessions.
💡
/clear doesn't lose your work

Context is not your code. When you /clear, the files you edited are still there — on disk, in git. You only lose the conversation history. The code Claude wrote survives the clear.

04 — Commands Reference

The commands you'll use every session.

Command / Key Shortcut What it does When to use
/init — Explores codebase and generates CLAUDE.md First time in a new project
/clear — Clears conversation history. Keeps CLAUDE.md context. Does not undo file edits. Between unrelated tasks, when Claude drifts, after 20–30 turns
/cost — Shows token usage and estimated cost for the current session Periodically during long sessions, end of session
/undo — Undoes the last set of file edits made by Claude Code When the last change was wrong and you want to revert immediately
Plan mode Shift+Tab Toggles plan-before-act mode. Claude shows the plan and waits for approval before editing any files. Any multi-file task, any time you want to see the approach before execution
Interrupt Ctrl+C Stops Claude Code mid-execution. Completed actions are not undone automatically. When Claude is heading in the wrong direction and you need to stop it
@filename — References a specific file, directory, or URL. Claude reads it before responding. Any task where you want Claude to know a specific file's contents
/help — Lists all available commands When you can't remember a command name
