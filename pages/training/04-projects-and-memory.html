Projects & Persistent Context

A Claude Project is a persistent workspace: a system prompt that runs on every conversation, shared files your team can reference, and usage visible across sessions. It is the difference between treating Claude as a search box and treating it as a team member with standing instructions.

Claude.ai · ~20 min
01 — What a Project is

A Project is a configured workspace, not just a folder.

When you create a Claude Project, three things persist across every conversation inside it: a system prompt you write once, any files you upload, and the conversation history. Individual chats are still separate — but the context they share is permanent.

Why this matters at team scale

Without a project system prompt, every team member either writes the full stack context in every message, or gets generic answers that assume the wrong stack. A shared project with a well-written system prompt means every teammate starts from the same baseline — and you don't have to train each person to write context-heavy prompts from scratch.

02 — Writing the system prompt

What belongs in a project system prompt — and what doesn't.

The system prompt runs on every single conversation, consuming tokens before you've said a word. Keep it dense with things that never change. Skip things that vary by task.

✓ Include

Permanent project context

  • Tech stack (framework, component library, build tool)
  • Compliance and accessibility constraints
  • User persona and domain vocabulary
  • Output preferences (code over prose, short answers, specific format)
  • Team conventions (naming patterns, PR conventions)
✗ Skip

Task-specific context

  • The specific feature you're working on right now
  • Which file you're editing
  • What you tried that didn't work
  • Temporary instructions that apply to one conversation
  • Examples that only apply to one task (use few-shot in the message instead)
03 — Project Files

Upload once. Reference in every conversation.

Files uploaded to a project are available in every chat — but Claude doesn't automatically read them. You reference them explicitly in your message. The advantage: you don't have to re-paste the same spec or schema in every conversation.

📄
Specs and requirements
Upload your PRD, component spec, or acceptance criteria document. Reference it with "Using the requirements in component-library.md, generate…" instead of pasting the whole thing inline.
🗺
Architecture docs
Data schema, API contracts, system diagrams. Claude can answer questions about your data model when the schema is in the project. Useful for generating mock data, validators, and typed interfaces.
🎨
Design tokens / style guide
If your team has a token list or brand guidelines doc, upload it. Claude will reference your actual color variable names instead of guessing at them.
⚠️
File token cost

Files are re-sent on every message in the conversation, not once. A 10-page spec PDF can add 3,000–5,000 tokens per turn. Keep project files to documents you actually reference often — don't upload everything you have.

04 — Usage patterns

How real teams structure their Claude projects.

One big shared project or many specialized ones? Both work — the right answer depends on how specialized your system prompt needs to be per role or workstream.

A
One project per team
Best when the team is small and the stack is consistent. The system prompt covers all roles. Everyone sees the same conversations. Simple to manage.
B
One project per role
A Design project with a UX-focused system prompt, a Dev project with an Angular implementation focus. Useful when designers and devs ask very different types of questions. More setup but more precise answers per role.
C
One project per feature/sprint
Temporary projects scoped to a specific feature. System prompt has the feature brief; files have the spec. Archive after the sprint. Keeps conversations on-topic and context window clean.
