Git, Commits & Review

Claude Code can write, stage, and commit code — but you own the git history. This lesson covers how to read Claude-generated diffs, what to audit before you commit, how to write commit messages that describe intent, and when to use Claude Code's built-in git commands versus doing it yourself.

Claude Code · ~20 min
01 — Reading Claude's Output

The diff is the most important step.

Claude Code shows you a diff of every change it makes. Most developers glance at it and commit. That's the wrong habit. The diff is your review — the same review you'd do on a junior developer's PR. Read it with the same care.

⚖️

Added vs. removed lines

A feature addition should mostly add lines. If Claude is removing a lot, read every removed line. Deleted code is usually intentional — but make sure you understand what you're losing.

🏗️

Pattern consistency

Does the new code match your existing style? Does it use the same variable names, the same patterns, the same error handling approach? Inconsistency now means technical debt later.

02 — The Pre-Commit Audit

Five things to check before every commit.

✓
Does it do what you asked?
Open the browser. Click the thing. Does it work? Don't skip this. Claude can write syntactically correct code that doesn't do what you described.
High — always check
✓
No hardcoded data or credentials
Scan the diff for strings that look like real data: email addresses, names, IDs, API keys. Claude sometimes uses realistic-looking placeholder data. In a financial services context, this matters.
High — always check
✓
No scope creep
Did Claude change files you didn't ask it to touch? Refactor something while adding a feature? These changes can be correct and still be wrong for this commit.
Medium — check on large tasks
✓
Accessibility compliance
Status badges have icon + label (not color alone)? Buttons have aria-label? New interactive elements are keyboard-navigable? WCAG 2.1 AA is in your CLAUDE.md, but verify in the diff.
Medium — check on UI changes
✓
Commit message describes intent, not mechanics
Claude-generated messages often describe what changed ("Updated dashboard.js"). Good messages describe why ("Add status filter to support Ahead/On-Track/At-Risk view"). Edit before committing.
Low — but affects future maintainability
03 — Commit Messages

Claude writes commits. You own the history.

Claude Code generates commit messages that describe what it did. These tend to be accurate but mechanical. A good commit message describes the intent — what behavior changed from the user's perspective — not just the implementation details.

What Claude generates Updated dashboard.js to add filterByStatus function and modified index.html to add three button elements with onclick handlers

Accurate. Mechanical. Describes implementation. Doesn't tell a future developer why this change existed or what user-facing behavior changed.

What you should commit feat: add status filter bar to portfolio table Adds Ahead / On Track / At Risk filter buttons above the loan table. Active filter highlights. Defaults to showing all branches (no filter).

Describes the feature. Includes the user-visible behavior. A developer reading this in 6 months understands what changed and why — without reading the diff.

✍️
Ask Claude to write it in conventional commits format

Prompt: "Write a commit message for this change in conventional commits format (feat/fix/chore/refactor). Describe the user-visible behavior change in the body." Claude follows the format well when told to.

04 — Claude Code vs. DIY Git

When to let Claude handle git, and when not to.

Let Claude do it
  • Add and commit after a clean feature
  • Write the initial commit message
  • Stage specific files you specify
  • Show git log / git status
  • Create a branch for a new feature
Do it yourself
  • Push to origin / remote branches
  • Rebase or amend published commits
  • Merge pull requests
  • Any git operation on shared history
  • Any force-push operation
🚫
Never let Claude push to a shared branch

Claude Code can run git push if asked. Don't ask it to. Commits on local branches are easy to undo. Commits pushed to origin and seen by the team are not. Own your push step manually.
