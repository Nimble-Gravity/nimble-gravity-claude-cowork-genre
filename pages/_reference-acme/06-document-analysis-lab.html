PRD Excerpt — LOAN-2024-Q3 Portfolio Dashboard 
Feature: Loan Application Review Table — Portfolio Dashboard

Overview:
The Q3 Portfolio Dashboard replaces the existing branch performance report with an
interactive table. The table is the primary workspace for loan officers reviewing
the LOAN-2024-Q3 portfolio across six branches.

Data Model:
Each row represents one branch with the following fields:
- Branch ID (string, e.g. "BR-001")
- Branch Name (string)
- Target Approvals (integer)
- Actual Approvals (integer)
- % to Target (calculated: Actual / Target × 100, rounded to 1 decimal)
- Last Updated (date, format: YYYY-MM-DD)
- Status (enum: Ahead | On Track | At Risk)

Status Logic:
- Ahead: Actual Approvals ≥ Target Approvals × 1.05
- On Track: Actual Approvals ≥ Target Approvals × 0.90
- At Risk: Actual Approvals < Target Approvals × 0.90

Functional Requirements:
1. The table must render all six branch rows on initial load.
2. Users must be able to filter rows by Status value.
3. The table must be sortable by any column header (ascending/descending).
4. Status must be communicated with a visual badge using both icon and label.

Non-Functional Requirements:
- Page load time under 2 seconds on standard corporate laptop hardware.
- Table must be keyboard navigable (WCAG 2.1 AA).
- Data must refresh when user manually triggers a refresh action.

Open Questions:
- Should the % to Target column use a progress bar or text value?
- What is the behavior when Last Updated is more than 30 days ago?

Analysis → Ticket → Gaps → Artifact.

Run these four prompts in one Project conversation. Each prompt builds on the previous one — do not start a new conversation between steps.

Prompt 2 — Generate a Jira-ready ticket

Turn the requirements from Step 1 into a complete, pasteable ticket description. Use this in the same conversation — Claude has the context from Prompt 1.

Prompt 2 — Jira Ticket 
Using the requirements you just extracted, write a Jira story for the Loan Application
Review Table feature.

Format:
**Summary:** [one-line story title]

**User Story:**
As a [persona], I want [capability] so that [benefit].

**Acceptance Criteria:**
[Gherkin format — one Scenario block per functional requirement]

**Definition of Done:**
- [ ] [checkbox items — implementation, tests, accessibility review, design sign-off]

**Open Questions:**
[from the PRD open questions section]

**Story Points:** [your estimate: 1, 2, 3, 5, 8, or 13]
**Labels:** [relevant labels from: frontend, accessibility, data, high-priority]
👁

Observe: Would you copy this directly into Jira?

What needs editing before this is actually usable? Is the user story persona correct for your team? Did the Gherkin scenarios match the requirements exactly? Note the gap between "usable immediately" and "needs 5 minutes of editing."

Step 3 — Find gaps
3

Prompt 3 — Identify coverage gaps

Ask Claude to find what's missing — a different kind of analysis than extraction.

Prompt 3 — Gap Analysis 
Review the requirements and ticket from our conversation. Identify:

1. **Missing edge case coverage** — scenarios that are not handled in the current requirements
   (e.g., empty states, error states, data edge cases)

2. **Accessibility gaps** — WCAG 2.1 AA requirements that are implied by the PRD but
   not explicitly specified as testable criteria

3. **Compliance concerns** — anything in the data handling or display that might
   require review for SOC 2 or PCI-DSS given that this is financial services data

Format each finding as:
**Gap:** [description]
**Why it matters:** [one sentence]
**Suggested addition:** [requirement text to add to the story]
👁

Observe: Did Claude find anything your team would have caught in review?

This is the value of early AI analysis — finding the "what about when the data is empty?" class of gap before sprint planning, not during QA. Compare against what a manual requirements review would typically catch.

Debrief

Where in your workflow does this fit?

"Which step would you add to your team's existing review process?"
Not replace — add. Gap analysis in Prompt 3 could happen before sprint planning. Ticket generation could happen before a requirements review meeting. Prototype could replace a low-fidelity wireframe in a stakeholder sync.
"What did Claude get wrong, and how long would it take to fix?"
Every output from Claude requires a review pass. Assess: how long did the editing take vs. the time it would have taken to produce the same output from scratch? What's the net time savings?
"What would you put in the Project files to improve Prompts 2 and 3?"
If you uploaded your Jira ticket template, could Prompt 2 produce a better-formatted ticket? If you uploaded your WCAG audit checklist, would Prompt 3 find more specific gaps?
