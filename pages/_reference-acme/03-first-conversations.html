First Conversations

Five prompts in claude.ai. Each one adds a layer of structure. Watch the output change. This is 20 minutes of hands-on work — open claude.ai in another tab now.

Hands-on Exercise · ~20 min
Exercise Goal

See how adding structure changes output quality.

You'll run the same underlying task five times, adding one structural element with each prompt. By Prompt 5, the output should be directly usable in a work context. By Prompt 1, it probably won't be.

📋
Before you start

Open claude.ai in a new tab. Use the free tier (Claude Sonnet) — these prompts are well within its capabilities. Start a new conversation for each prompt so context doesn't carry over between them. Don't reuse a single conversation for all five.

The Five Prompts

Run each prompt. Compare the outputs.

Module 1 · Hands-On Exercise
Five Structured Conversations
~20 min

Copy each prompt into a new claude.ai conversation. Do not edit the prompts first — run them exactly as written, then assess the output before moving on.

1

Prompt 1 — No structure (the baseline)

Start here to anchor your expectations. This is how most people prompt Claude when they first start.

Copy into a new claude.ai chat →
Prompt 1 — Baseline 
Write acceptance criteria for a loan status feature.
👁

Observe: What did Claude assume?

Notice the tech stack it assumed. The status values it invented. The format it chose. The level of specificity. This is Claude's best guess with zero context. How much of it is usable in your actual project?

Add: Task clarity
2

Prompt 2 — Add a specific task

Same subject. Now we tell Claude what kind of output we want and name the actual values.

Prompt 2 — Specific task 
Write Gherkin-format acceptance criteria for a status badge feature.

The badge must display one of three values: Ahead, On Track, or At Risk.
👁

Observe: Did Gherkin format appear?

Did specifying Gherkin improve the structure? Note whether the three status values are used. Notice if Claude still invented context you didn't provide.

Add: Role + context
3

Prompt 3 — Add role and project context

Now Claude knows who it is and what the codebase looks like. This is where outputs start becoming project-specific.

Prompt 3 — Role + context 
You are a QA engineer writing acceptance criteria for a financial services app.

The app is built with forked Angular Material. Users are loan officers and credit analysts
who review loan application portfolios. WCAG 2.1 AA compliance is required.

Write Gherkin-format acceptance criteria for a status badge feature.
The badge must display one of three values: Ahead, On Track, or At Risk.
Status must never be communicated by color alone — each badge needs an icon + label.
👁

Observe: Did the accessibility constraint appear?

Check whether the icon + label requirement made it into the criteria. Notice whether "loan officer" changed any of the language. Compare specificity to Prompt 2.

Add: Output format
4

Prompt 4 — Add output format constraint

Specifying exactly how many criteria, and that edge cases must be covered, forces Claude to be exhaustive rather than illustrative.

Prompt 4 — Full structure 
You are a QA engineer writing acceptance criteria for a financial services app.

The app is built with forked Angular Material. Users are loan officers and credit analysts
who review loan application portfolios. WCAG 2.1 AA compliance is required.

Write Gherkin-format acceptance criteria for a status badge feature.
The badge must display one of three values: Ahead, On Track, or At Risk.
Status must never be communicated by color alone — each badge needs an icon + label.

Output format:
- One Scenario block per status value (3 total)
- One Scenario for an unknown/missing status (edge case)
- Use GIVEN / WHEN / THEN structure
- Keep each scenario to 5 lines maximum
👁

Observe: Did the edge case scenario appear?

Count the scenarios. Did Claude hit exactly 4? Did specifying 5-line maximum keep responses concise? Is this output directly pasteable into your test management system?

Add: Few-shot example
5

Prompt 5 — Add a few-shot example

The most powerful technique for consistent output. One example tells Claude more about format than 200 words of description.

Prompt 5 — Few-shot 
You are a QA engineer writing acceptance criteria for a financial services app.
The app is built with forked Angular Material. Users are loan officers and credit analysts.
WCAG 2.1 AA — status must never be communicated by color alone.

Follow this exact format for each scenario:

---
Scenario: [Status value] badge renders correctly
  GIVEN a loan officer is viewing the LOAN-2024-Q3 portfolio table
  WHEN a row has status "[status value]"
  THEN the badge displays a [icon description] icon AND the label "[status value]"
  AND the badge color is [color] with sufficient contrast ratio (4.5:1 minimum)
---

Generate scenarios for: Ahead, On Track, At Risk, and Unknown status.
👁

Observe: Did the format hold across all four scenarios?

Compare the structure of each scenario. Did Claude follow the template exactly? Did it infer reasonable icon descriptions? Is the contrast ratio language present in every scenario? This is what usable output looks like.

Debrief

Three questions to anchor the lesson.

Discuss with your group or write down your answers. There are no right answers — the goal is to notice what changed across the five prompts.

"Which prompt would you use in production, and why?"
Consider: what's the minimum structure needed to get a usable output the first time? Is Prompt 5 always better than Prompt 4, or does it depend on the task?
"At what point did Claude stop guessing about your stack?"
Prompt 1 had to invent everything. When did Angular Material, the user role, and the accessibility requirement first appear in the output? Was it always when you mentioned them?
"What would you add to Prompt 5 for your actual project?"
Your project has a specific ALM tool, component naming conventions, and team standards. What 2–3 pieces of context would make these criteria directly usable without editing?
