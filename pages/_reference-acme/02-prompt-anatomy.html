Prompt Anatomy

A prompt is not a search query. It is a specification. Four components — Role, Task, Context, Format — determine whether Claude gives you what you need or what it guesses you might want.

Foundations · ~20 min
01 — The four components

Role, Task, Context, Format. In that order.

You don't need all four every time. But when you need a specific, reliable result — especially on complex tasks — structure beats free-text every time. Here's what each component does.

Role
Who Claude is
Sets the mental model and vocabulary Claude should apply. Not "act like" theater — a genuine instruction about what expertise to draw from.
"You are a senior UX engineer reviewing a component spec."
Task
What to do
The explicit action. One main verb. Avoid bundling multiple tasks in one sentence — Claude will prioritize the first one and shortchange the rest.
"Review this PRD section and identify missing acceptance criteria."
Context
What Claude needs to know
Background Claude can't infer: your tech stack, constraints, audience, what you've already tried. The more specific, the less Claude guesses.
"We use forked Angular Material. WCAG 2.1 AA is required. Our users are loan officers, not consumers."
Format
Shape of the output
How you want the answer structured. Specifying format removes ambiguity and reduces the chance of getting prose when you needed a table — or a table when you needed a paragraph.
"Return a numbered list. Each item: requirement text, then a 1-sentence gap description."
02 — Before and After

Same task. Completely different results.

The prompt on the left will get you a generic answer. The prompt on the right will get you something usable on the first try. The difference is ~30 words of structure.

Weak prompt
User prompt 
Write a status badge component.

What Claude gives you: A generic React component with color-only status indicators. No role. No tech stack. No accessibility constraints. No output format. Claude guesses everything — and guesses wrong because it defaults to the most common context on the internet, which is not yours.

Structured prompt
Structured prompt 
You are a front-end developer working on a forked Angular Material component library.

Create a StatusBadge component that shows loan application status.

Context:
- Status values: Ahead | On Track | At Risk
- WCAG 2.1 AA — never use color alone; each status needs an icon + label
- No Tailwind; use Angular Material theming tokens
- Users are loan officers on desktop (1280px+)

Return:
1. Component HTML template
2. Component TypeScript class
3. A brief note on the accessibility pattern used

What Claude gives you: An Angular Material component with icon + label badges, correct theming token usage, and an explicit ARIA explanation. Usable on the first pass because Claude had no guessing to do.

03 — System prompts

System prompts set the permanent context.

In Claude's Projects feature and in the API, a system prompt runs before every user message. It's the right place for role, constraints, and context that never change — so you don't have to repeat them in every prompt.

System prompt — set once

What belongs in a system prompt

Role + permanent context + standing constraints. Example: "You are a senior engineer on the Acme Corp loan origination platform. The component library is forked Angular Material. All UI must meet WCAG 2.1 AA. Respond concisely; prefer code over prose."

Set this in a Project system prompt and every conversation inherits it automatically.
User prompt — per task

What belongs in each message

Task + task-specific context + output format. You don't need to re-explain who you are or what stack you use if the system prompt covers it. Keep user messages focused on the specific thing you want done right now.

Shorter user messages + a good system prompt = more precise answers at lower cost.
04 — Few-shot examples

Show Claude one example and it will match your pattern exactly.

Few-shot prompting means providing 1–3 examples of the output format you want before asking Claude to generate new instances. It is the single most effective technique for consistent, structured output.

Few-shot prompt 
Convert each requirement into a Jira-ready acceptance criterion.
Follow this exact format:

Example input:
The table must show loan status.

Example output:
GIVEN a loan officer views the LOAN-2024-Q3 portfolio table
WHEN the page loads
THEN each row displays a Status badge with one of three values:
  Ahead | On Track | At Risk

---
Now convert:
The table must support filtering by status.
🎯
When to use few-shot

Any time you need consistent structure across many outputs — requirement extraction, ticket generation, changelog entries, test case format. One good example beats a 200-word format description every time.
